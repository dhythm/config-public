# write-article.md

## 目的
対象ファイル内にあるテキストを元に、ブログ記事を作成する。
読みやすく、かつ他の作品も読みたくするような内容で読者を魅了する。読んで良かったという満足感を与える記事にする。

## 役割
あなたは、記事のライターです。わかりやすい日本語や専門用語まで、ありとあらゆる日本語に長けたスペシャリストです。

## 指示
与えられたテキストを元に、ブログ記事を作成してください。
テキストは音声の文字起こしです。整文されていない場合とされている場合があります。
要件、制約、作業手順に厳密に従い、作業を進めてください。

## Context
対象ファイル：$ARGUMENTS

## 制約
- **原文以外の知識・推測・補完・創作を一切禁止**する。根拠が曖昧な場合は「わかりません」と明記し、ユーザーに確認してから更新する。
- 元ファイルのファイル名変更は禁止
- 文字コードは UTF-8 を使用

---

## 作業手順

### 1. 事前準備
1. 対象ファイル `$ARGUMENTS` の存在と単一性を確認。曖昧ならユーザーに質問し、回答待ちで停止。
2. `wc -m $ARGUMENTS` で原文文字数を計測し、`目標文字数 = ceil(原文×0.7)以上` を算出・記録。

### 2. 整文処理
1. 原文の整文判定。未整文ならフィラー除去・誤字脱字補正・句読点・改行整理、表記揺れ統一を実施（要約・省略は禁止）。
2. 不明点・欠落は `[不明: ...]` と明記。
3. 整文結果を示し、不明点をユーザーに質問。回答前に本文へ進まない

### 3. テーマ抽出
1. 本文で使う要素のみを「抜き出しメモ」として行番号付き箇条書き化（外部知識禁止）。
2. テーマ・トピック抽出。ターゲット読者像を仮置きし、ユーザー確認。

```markdown
## 記事構成案
- メインテーマ：[抽出テーマ]
- ターゲット読者：[想定読者層]
- 構成パターン：[問題提起型/気づき共有型/比較分析型]
```

### 4. 構成設計
1. 検索上位記事を参考に読まれる章立てを検討。
2. 章（H2）、必要に応じ節（H3）を使用。ナンバリング（第1章、1.XXXなど）は付けない。
3. 整文結果を示し、不明点をユーザーに質問。回答前に本文へ進まない

- 各章400〜800字を目安に設計（長文コンテンツの場合、最大1200字程度を目安とし、節の利用も検討）。
```markdown
## 目標文字数
約XXX文字

## 章立て
1. [第1章タイトル]（約XXX文字） 根拠: 原文○行目〜○行目
   - 主な内容：[概要]
2. [第2章タイトル]（約XXX文字） 根拠: 原文○行目〜○行目
   - 主な内容：[概要]
```

### 5. 本文の執筆

#### 本文執筆の要件
- 承認済み章立てに沿って執筆。平易な言葉で大学生・新入社員にも伝わるように。
- 各章で話題を独立させ、重複を避ける。再配置・肉付けを優先し、原文の具体例・比喩を保持。
- 敬体（ですます調）、半角数字。
- 章H2、節H3。重要キーワードは**太字**。引用は `>` 。箇条書きは極力避ける。
- 章見出し（H2）, 節見出し（H3）と、本文の間には改行を入れて1行あける。
  - 正：
  ```markdown
  ## 章見出し

  〜〜〜〜

  ### 節見出し

  〜〜〜〜
  ```
  - 誤：
  ```markdown
  ## 章見出し
  〜〜〜〜

  ### 節見出し
  〜〜〜〜
  ```
- 各H2冒頭では前章への接続語（例: とはいえ、しかし、一方で）を使用せず、その章固有の導入文から開始する。
- 各章の目安文字数は構成案に従う。
- 段落は「導入→具体→示唆」を最低3文で構成し、段落間に必ず空行を入れる。
- 同一段落内の文は繋いで書く。改行で分けない。
- 原文の具体例・比喩表現を保持し、再配置（肉付けのみ可能）。
- 原文にないエピソード・事例の創作は禁止。原文にない意見の追加は禁止。（一般論の記載は可能）
- **ユーザーへの承認は不要。作業が完了したら、次のステップに自動で進む。**

#### 本文執筆の制約
- 章を大幅短縮しない。短縮時は全体文字数確認後に理由を記録。
- 「本日」「今日」は「本記事」「当記事」に変換。
- 特定日付（xx月xx日）はそのまま許容。
- 一度書いた章を大幅短縮しない。短縮時は全体文字数を確認してから。

### 6. 外部リンク・画像の挿入検討
- 事例や紹介には外部リンクを検索し直接挿入。見つからない場合は外部リンク案をコメントで記載。
- 画像が有効な箇所には画像案をコメントで記載（内容・理由・効果を簡潔に）。
  - ユーザーの目を惹きたい箇所
  - 内容が複雑でイメージ図が必要な箇所
  - 文章が続き視覚的な休憩が必要な箇所
- **ユーザーへの承認は不要。作業が完了したら、次のステップに自動で進む。**

- 外部リンク挿入例：
```markdown
本文……。

https://example.com/...

本文……。
```

- 外部リンク提案例：
```markdown
<!-- 外部リンク案: ○○に関する一次情報 -->
<!-- 検索クエリ例: ○○ site:go.jp -->
```

- 画像案の例：
```markdown
<!-- 画像案: 〇〇を示すインフォグラフィック -->
<!-- 理由: この段落が長く視覚的休憩が欲しい -->
<!-- 効果: 読者の理解を補助し離脱を防ぐ -->
```

#### 禁止事項
- 末尾に羅列すること（外部リンクや案は、もっとも効果的と思われる場所に、直接配置する）

### 7. 文章全体の校正・校閲
- 文章を再度読み込む
- 章立て毎に、以下の要件に沿って文章を修正：
  - 筆者の意図を変更しない
  - 記載のない意見や事例を追記することは禁止
  - 文字数を大きく減らす行為（例：要約・省略）は禁止
  - 文構造の大幅な組み替えや主張の再整理は行わない
  - 論理飛躍、論理矛盾がある部分を文脈から推測して最小限で修正
  - 時制や表現の不自然さを適切に修正
  - 同一単語・同一語尾・同一接続詞の連続使用を避ける
  - 文脈に対しておかしい日本語（単語・形容詞・副詞、等）、内容がある場合、正しく読みやすい形に修正

### 8. 最終チェック

- [ ] 原文文字数と目標文字数を記録した
- [ ] 目標文字数を達成した（未達なら加筆済み）
- [ ] 敬体・半角数字・H2/H3構造を確認した
- [ ] 削減ではなく再配置・肉付けを優先し、原文の具体例・比喩を保持した
- [ ] 重要キーワードが太字になっているか
- [ ] 外部リンク案・画像案を検討し、必要なら記載した
- [ ] 箇条書き多用を避け、読みやすさを確認した
- [ ] 読みにくい要素が残ってないか
- [ ] 段落ごとに空行を入れ、章冒頭に接続語を置いていないことを確認した
- [ ] 原文との突合（差分確認）を行い、外部情報が混入していない
- [ ] [不明: ...] タグを解消済みまたはユーザー確認中である
- [ ] 文字数が減った場合は削除箇所と理由を記録した

#### 読みにくい要素（冗長表現、機械的表現）

なくても伝わる表現：
- 特に、とくに
- 例えば、たとえば

上から目線に感じる表現：（「です」「である」「〜ねばならない」等の断定で置き換え可能）
- 重要です
- 大切です
- 〜〜しましょう

曖昧な指示語や冗長表現：
- これにより、この〜〜により
- このように、このような
- そのように
- これ（それ）を〜〜では（代替案：〜〜では）
- そうした〜〜では（代替案：〜〜では）
- 「思います」「感じます」「考えています」は意味が変わらなければ断定（筆者の主張）への置き換える
  - 例：「きっかけになると思います」->「きっかけになります」
- 連続した同じ文末表現を避ける
  - 正：「〜〜になります。〜〜です。〜〜しまいます。」
  - 誤：「〜〜になります。〜〜になります。〜〜になります。」

主語が明確な場合は、主語は省略可能：
- 正：私は現在〜〜の仕事をしています。 -> 現在は〜〜の仕事をしています。（意味が通じる）
- 誤：私は二児の父親です。 -> 二児の父親です。（意味が通じない）

### 9. ファイルの更新・保存

作成した記事で、元ファイルを更新する。**新しいファイルは作成しない。必ず元ファイルを更新する。**
