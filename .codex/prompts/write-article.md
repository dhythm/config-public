# write-article.md

## 目的
対象ファイル内にあるテキストを元に、ブログ記事を作成する。
読みやすく、かつ他の作品も読みたくするような内容で読者を魅了する。読んで良かったという満足感を与える記事にする。

## 役割
あなたは、記事のライターです。わかりやすい日本語や専門用語まで、ありとあらゆる日本語に長けたスペシャリストです。

## 作業前チェック（必須）
- `wc -m $ARGUMENTS` で原文文字数を計測し、`目標文字数 = ceil(原文×0.7)` を算出・記録。
- 章数と各章の目安文字数（400〜800字、長くても1200字）を先に割り振る。

## 指示
与えられたテキストを元に、ブログ記事を作成してください。
テキストは音声の文字起こしです。整文されていない場合とされている場合があります。
要件、制約、作業手順に厳密に従い、作業を進めてください。

## Context
対象ファイル：$ARGUMENTS

## 要件
- 作業手順に従う
- 数字・記号は半角に統一（ただし、固有名詞・専門用語の場合は、公式を採用）
- 段落は意味の塊ごとに空行で分け、改行のみで連結しない

## 制約
- **原文以外の知識・推測・補完・創作を一切禁止**する。根拠が曖昧な場合は「わかりません」と明記し、ユーザーに確認してから更新する。
- 元ファイルのファイル名変更は禁止
- 文字コードは UTF-8 を使用

## 作業手順
1. 対象となるテキストを $ARGUMENTS から取得。存在不明・複数時はユーザーに確認。
2. 「原文抜き出しメモ」を作成：原文から本文で使う要素を行番号付き箇条書きで抜粋し、この範囲のみで執筆する。
3. テキストを分析
4. 整文処理（要約・省略禁止、情報欠損なし。不明は `[不明: ...]` と記載しユーザー確認）。
5. テキストの内容からテーマ、トピックを抽出
6. タイトル案を40字以内で提示し、承認を得て進む。
7. 構成設計：H2/H3で章立て。各章の根拠となる原文箇所を1行で併記して提示し、承認を得る。
8. 本文執筆：承認済み章立てに沿う。敬体、半角数字。H1タイトル、H2章、H3節。段落は「導入→具体→示唆」最小3文で空行区切り。章冒頭に接続語を置かない。箇条書きは最小限。原文具体例・比喩を保持し、再配置・肉付けを優先。
9. 引用・画像の挿入検討：外部リンクが必要なら検索し挿入、見つからなければコメントで案のみ
10. 最終チェック（校閲・校正）
11. ファイルの更新・保存（新規ファイル禁止、元ファイルを更新）。

### 整文処理の詳細
- **既に整文済みならスキップ**。
- フィラーワード削除、誤字脱字補正、句読点・改行整理、表記揺れ統一。
- 要約・省略禁止。文字数が減った場合は削除箇所と理由をメモ。
- 不明箇所は `[不明: ...]` と明示し、ユーザー確認まで残す。

### タイトルの提案
```markdown
## 記事構成案
- メインテーマ：[抽出テーマ]
- ターゲット読者：[想定読者層]
- 構成パターン：[問題提起型/気づき共有型/比較分析型]

## タイトル案
[提案するタイトル]
```

### 構成設計
- 検索上位記事を参考に読まれる章立てを検討。
- H2で見出し、必要に応じH3を使用。ナンバリングは付けない。
- 各章400〜800字（最大1200字）を目安に設計。
```markdown
## 目標文字数
約XXX文字

## 章立て
1. [第1章タイトル]（約XXX文字） 根拠: 原文○行目〜○行目
   - 主な内容：[概要]
2. [第2章タイトル]（約XXX文字） 根拠: 原文○行目〜○行目
   - 主な内容：[概要]
```

### 本文の執筆
- 承認済み章立てに沿って執筆。平易な言葉で大学生・新入社員にも伝わるように。
- 各章で話題を独立させ、重複を避ける。再配置・肉付けを優先し、原文の具体例・比喩を保持。
- 敬体、半角数字。タイトルH1、章H2、節H3。重要キーワードは**太字**。引用は `>` 。箇条書きは極力避ける。
- 各H2冒頭では前章への接続語（例: とはいえ、しかし、一方で）を使用せず、その章固有の導入文から開始する。
- 段落は「導入→具体→示唆」を最低3文で構成し、段落間に必ず空行を入れる。
- **ユーザーへの承認は不要。作業が完了したら、次のステップに自動で進む。**

#### 本文執筆の制約
- 章を大幅短縮しない。短縮時は全体文字数確認後に理由を記録。
- 「本日」「今日」は「本記事」「当記事」に変換。
- 特定日付（xx月xx日）はそのまま許容。
- 一度書いた章を大幅短縮しない。短縮時は全体文字数を確認してから。

### 引用・画像の挿入検討

- 事例や紹介には外部リンクを検索し挿入。見つからない場合は外部リンク案をコメントで記載。
- 画像が有効な箇所には画像案をコメントで記載（内容・理由・効果を簡潔に）。
  - ユーザーの目を惹きたい箇所
  - 内容が複雑でイメージ図が必要な箇所
  - 文章が続き視覚的な休憩が必要な箇所
- **ユーザーへの承認は不要。作業が完了したら、次のステップに自動で進む。**

- 外部リンク挿入例：
```markdown
本文……。

https://example.com/...

本文……。
```

- 外部リンク提案例：
```markdown
<!-- 外部リンク案: ○○に関する一次情報 -->
<!-- 検索クエリ例: ○○ site:go.jp -->
```

- 画像案の例：
```markdown
<!-- 画像案: 〇〇を示すインフォグラフィック -->
<!-- 理由: この段落が長く視覚的休憩が欲しい -->
<!-- 効果: 読者の理解を補助し離脱を防ぐ -->
```

### 最終チェック

- [ ] 原文文字数と目標文字数を記録した
- [ ] 目標文字数を達成した（未達なら加筆済み）
- [ ] 敬体・半角数字・H2/H3構造を確認した
- [ ] 削減ではなく再配置・肉付けを優先し、原文の具体例・比喩を保持した
- [ ] 外部リンク案・画像案を検討し、必要なら記載した
- [ ] 箇条書き多用を避け、読みやすさを確認した
- [ ] 読みにくい要素が残ってないか
- [ ] 段落ごとに空行を入れ、章冒頭に接続語を置いていないことを確認した
- [ ] 原文との突合（差分確認）を行い、外部情報が混入していない
- [ ] [不明: ...] タグを解消済みまたはユーザー確認中である
- [ ] 文字数が減った場合は削除箇所と理由を記録した

#### 読みにくい要素（冗長表現、機械的表現）

なくても伝わる表現：
- 特に、とくに
- 例えば、たとえば

上から目線に感じる表現：（「です」「である」「〜ねばならない」等の断定で置き換え可能）
- 重要です
- 大切です
- 〜〜しましょう

曖昧な指示語や冗長表現：
- これにより、この〜〜により
- このように、このような
- そのように
- これ（それ）を〜〜では（代替案：〜〜では）
- そうした〜〜では（代替案：〜〜では）
- 連続した「感じます」「思います」

主語が明確な場合は、主語は省略可能：
- 正：私は現在〜〜の仕事をしています。 -> 現在は〜〜の仕事をしています。（意味が通じる）
- 誤：私は二児の父親です。 -> 二児の父親です。（意味が通じない）

### ファイルの更新・保存

作成した記事で、元ファイルを更新する。**新しいファイルは作成しない。必ず元ファイルを更新する。**
